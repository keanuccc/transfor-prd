# 项目经历：TransforPRD — AI 驱动的产品需求文档生成工具

## 项目概述

**TransforPRD** 是一款纯前端 AI 应用，支持用户上传 Markdown 思维导图，调用大语言模型（LLM）一键生成结构化的产品需求文档（PRD）。项目支持多模型切换、流式生成、分段续写、竞赛对比、版本快照、知识库注入、加密分享、平台同步等高级功能，覆盖 PRD 从构思到分发的完整链路。项目已开源发布在 GitHub。

- **项目类型**：个人项目（独立开发）
- **项目周期**：约 3 个月
- **技术栈**：React 19 + TypeScript + Vite 8 + Tailwind CSS 4 + Zustand + Dexie.js + Web Crypto API
- **代码规模**：~90 个源文件，包含完整的单元测试、组件测试
- **GitHub**：[https://github.com/keanuccc/transfor-prd](https://github.com/keanuccc/transfor-prd)

## 项目背景

在互联网产品开发流程中，产品需求文档（PRD）是连接需求与研发的关键纽带。传统 PRD 写作存在三个痛点：

1. **格式繁琐**：PRD 有固定的章节结构和内容要求，手动撰写耗时且容易遗漏关键信息
2. **思维导图无法直接落地**：产品经理常用思维导图梳理思路，但从思维导图到结构化 PRD 之间存在巨大鸿沟
3. **迭代效率低**：需求变更后需要重新修订文档，缺少版本回溯和对比手段

市面上已有 Notion AI、腾讯文档 AI 等通用 AI 写作工具，但尚无专门针对「思维导图→PRD」这一场景的垂直工具。本项目填補了这一空白。

### 技术选型的核心原则

项目始终贯彻「轻后端、重前端」的架构理念 — **零后端服务，所有逻辑在浏览器中完成**。这一决策基于以下考量：

- **隐私优先**：用户的 API Key 和文档数据全程不经过第三方服务器
- **零部署成本**：无需服务器，无需数据库，静态托管即可上线
- **离线可用**：Service Worker 缓存后，核心功能在离线状态下仍可使用

### 第一阶段：MVP 原型验证（第 1-2 周）

- 搭建 React + Vite + Tailwind 项目脚手架
- 实现最小可用链路：上传 .md 文件 → 拼接 prompt → 调用 LLM API → 渲染 Markdown 结果
- 验证核心假设：思维导图格式能被 LLM 正确理解并生成有用 PRD

**产出**：可演示的原型，验证了产品方向可行。

### 第二阶段：核心体验打磨（第 3-5 周）

- 引入 Zustand 状态管理 + Dexie.js 持久化，实现对话记录的增删改查
- 实现 SSE 流式输出，解决长文档生成时的页面无响应问题
- 解决 LLM 输出截断问题：实现 `finish_reason: length` 自动续写机制
- 引入 AbortController 实现中途停止生成

**关键攻克点**：流式输出的性能优化。初始版本每个 chunk 触发一次 React 重渲染，GPU 占用率高达 80%。通过 rAF 批处理将更新频率从每毫秒降低到每帧（~16ms），渲染性能提升 5 倍以上。

### 第三阶段：高级功能实现（第 6-9 周）

- **澄清对话**：生成前 LLM 提出 2-3 个问题，用户回答后注入 prompt 提升 PRD 质量

- **分段生成**：支持思维导图选择性输入，用户可勾选特定分支生成对应章节
- **精修 / 审阅 / 评分**：对已生成 PRD 追加 AI 操作，每个操作产生独立消息标签页
- **代码骨架生成**：基于 PRD 描述自动生成项目目录结构和空文件
- **版本快照**：自动/手动保存 PRD 历史版本，支持一键恢复
- **知识库注入**：自定义知识文档，生成时注入为额外上下文

### 数据流核心链路

```
用户上传思维导图 (.md)
  → HomePage 解析文件内容
  → conversationStore.createConversation() 写入 IndexedDB
  → 导航至 RunPage
  → useEffect 触发 useLLMStream.startGeneration()
  → ping 模型连通性（10s 超时）
  → 构建 System Prompt（模板 + 知识库 + 澄清 Q&A）
  → streamChatCompletion() 发起 SSE 请求
  → AsyncGenerator 逐 chunk yield
  → rAF 批量更新 conversationStore 中的消息 content
  → MarkdownPreview 实时渲染
  → 遇到 finish_reason: length → 自动追加"请继续生成"
  → 生成完成 → 播放完成音效
```

### 持久化分层策略

| 存储层 | 技术 | 存储内容 | 设计理由 |
|--------|------|---------|---------|
| IndexedDB（加密） | Dexie + AES-GCM | 无（密钥库独立） | API Key 的加密密钥隔离存储 |
| localStorage（加密） | AES-GCM | LLM 配置（含 API Key） | 防止明文泄露 |
| localStorage（明文） | Zustand persist | 主题、设置、模板、知识库 | 配置项小，需同步读取 |
| IndexedDB（明文） | Dexie | 对话、消息、版本快照 | 数据量大，需索引查询和事务支持 |

## 关键设计决策

### 决策一：纯前端架构 vs. 引入后端服务

**问题**：用户数据（API Key、对话记录）应该存储在哪里？是否需要自建后端？

**分析**：
- 引入后端 → 需要用户系统、数据库、服务器运维，增加隐私风险和开发成本
- 纯前端 → 数据全部在浏览器，用户隐私得到保障，零部署成本，但存储容量和技术实现受限

**决策**：纯前端架构。理由：
1. API Key 涉及用户资金安全，不应经过第三方服务器，Web Crypto API 可提供客户端级加密
2. IndexedDB 存储上限远大于 localStorage（通常数百 MB），足够存储对话和文档
3. 配合 Service Worker 可实现离线使用，体验接近原生应用
4. 静态托管（GitHub Pages / Vercel）零成本部署

**权衡**：放弃了服务端渲染（SEO 不重要）、放弃了跨设备同步（未来可通过可选后端补上）。

### 决策二：Dexie.js 作为 IndexedDB 封装 vs. 直接操作 IndexedDB API

**问题**：原生 IndexedDB API 有大量回调式样板代码，如何高效管理对话数据的增删改查？

**分析**：
- 原生 IndexedDB → 无额外依赖，但 API 设计老旧，事务、索引、版本迁移都需大量代码
- Dexie.js → 约 15KB gzip，Promise-based API，内置版本迁移和复合索引支持

**决策**：选用 Dexie.js。版本迁移示例：

```typescript
// v1: 初始化
db.version(1).stores({ conversations: 'id, createdAt, updatedAt', messages: 'id, conversationId, createdAt' })

// v2: 新增字段
db.version(2).stores({ conversations: 'id, createdAt, updatedAt, tags, starred', ... })
  .upgrade(tx => tx.table('conversations').toCollection().modify(c => { c.tags = []; c.starred = false }))

// v3: 新增快照表
db.version(3).stores({ snapshots: 'id, conversationId, createdAt' })
```

### 决策三：自动续写机制的实现方式

**问题**：LLM 的输出 token 上限（通常 4096-8192）不足以一次性生成完整 PRD，如何处理截断？

**方案对比**：
- 方案 A：预先估算 token 数，分段请求 → 需要精确的 tokenizer，且无法保证每段内容质量一致
- 方案 B：让 LLM 自己续写，遇到截断自动发送"请继续" → 无需 token 估算，依赖 LLM 理解上下文

**决策**：采用方案 B，通过 `finish_reason: length` 信号触发自动续写，限制最多 5 次续写。实现方式是将已生成的部分追加为 assistant 消息、再追加 "请继续生成" 的 user 消息，构建新的消息历史重新请求。这比 token 估算方案更稳定可靠。

### 决策四：流式渲染性能优化

**问题**：LLM 返回的 stream chunk 频率可高达每秒 50+ 次，每次触发 React setState → 重渲染 → DOM 更新会造成严重性能问题。

**解决方案**：实现 requestAnimationFrame 批处理队列：
- chunk 到达时写入缓冲区，不立即 setState
- pending rAF 标记防止重复注册回调
- 每帧刷新一次（~16ms），将缓冲区内容一次性合并写入状态
- 在无 rAF 的环境下降级为 80ms 间隔轮询

**效果**：渲染次数从 ~2000 次/分钟降至 ~60 次/分钟，GPU 占用从 80% 降至 <5%。

### 决策五：API Key 客户端加密方案

**问题**：用户的 LLM API Key 如何安全存储？

**方案设计**：
1. 首次使用时，Web Crypto API 生成 256 位 AES-GCM 密钥
2. 密钥原始字节存储于独立 IndexedDB 数据库（`transfor-prd-keystore`），与业务数据隔离
3. 所有 localStorage 中的 LLM 配置数据通过此密钥加密/解密
4. Zustand persist 中间件通过自定义 `createEncryptedStorage()` 适配器接入

**安全性边界**：此方案的安全边界是用户的设备。密钥存储在 IndexedDB 中，与 localStorage 的加密数据分离。攻击者需要同时获取 IndexedDB 数据 + localStorage 数据 + 知道解密逻辑才能恢复 API Key。主要防护的是浏览器扩展窃取、本地文件泄露等场景。

### 决策六：文档加密分享

**问题**：如何实现「无需服务端」的文档分享，同时保证只有知道密码的人能阅读？

**方案设计**：
1. 用户设定分享密码
2. PBKDF2（10 万次迭代，SHA-256）从密码派生 256 位 AES-GCM 密钥
3. 使用随机 salt（16 字节）+ 随机 IV（12 字节）加密 PRD 内容
4. 将 `{ salt, iv, data }` 编码为 Base64，通过 URL Hash 传递
5. `/share` 路由独立于主应用布局，接收方输入密码后客户端解密

**优势**：服务端无法解密（密码不出浏览器），链接可安全通过任意渠道传播。

## 项目成果

项目最终交付了一款功能完整的 AI 驱动 PRD 生成工具，核心功能包括：

### PRD 生成与编辑

- **思维导图转 PRD**：上传 Markdown 思维导图文件，一键生成结构化的产品需求文档，支持选择性输入（用户可勾选特定分支只生成对应章节）
- **多模板支持**：内置 PRD 文档、用户故事地图、技术规格文档三种模板，支持自定义模板，覆盖产品规划的主要文档场景
- **流式实时预览**：LLM 生成内容以 SSE 流式输出，PRD 内容实时渲染为 Markdown，用户可即时看到生成进度
- **自动续写**：当 LLM 输出达到 token 上限时自动追加「请继续生成」，最多 5 次续写，确保长文档完整输出
- **中途停止**：支持随时中止生成，保留已生成部分，不会丢失当前进度
- **Markdown 编辑器**：内嵌完整的 Markdown 编辑器工具栏，支持粗体、斜体、标题、列表、代码块、表格、链接插入，以及导出为 Word / PDF / HTML / LaTeX 格式

### AI 辅助优化

- **澄清对话**：生成前 AI 自动提出 2-3 个澄清问题，用户回答后 Q&A 注入 prompt，显著提升 PRD 质量和对需求的贴合度
- **精修（Refine）**：对已生成的 PRD 进行风格优化和内容润色
- **审阅（Review）**：AI 从完整性、清晰性、可实现性、可测试性四个维度对 PRD 进行结构化审阅，给出修改建议
- **评审打分**：AI 对 PRD 进行多维度评分，结果以雷达图可视化展示
- **反向导出思维导图**：将已生成的 PRD 反向转换为 Markdown 思维导图，便于再次编辑和迭代
- **代码骨架生成**：基于 PRD 描述自动生成项目目录结构和空文件，加速开发启动

### 多模型协作

- **多模型配置**：支持配置多个兼容 OpenAI 接口的 LLM（如 DeepSeek、通义千问、智谱 GLM 等），灵活切换生成模型
- **竞赛模式**：2-3 个模型同时生成同一主题 PRD，三列对比展示，支持投票评选最佳结果
- **模型对比**：一键切换模型对同一份 PRD 重新生成，Diff 对比差异

### 数据管理

- **对话记录持久化**：所有生成记录自动保存到 IndexedDB，支持对话搜索、收藏、标签分类
- **文件夹分组**：对话支持按文件夹管理，保持工作区整洁
- **版本快照**：每次精修或编辑前自动保存 PRD 快照，支持手动保存和随时恢复至任意历史版本
- **知识库注入**：支持自定义知识文档，在生成 PRD 时注入为额外上下文，让输出更符合业务规范

### 安全与分享

- **API Key 加密存储**：基于 Web Crypto API（AES-GCM）对 API Key 进行客户端加密，密钥隔离存储，防止明文泄露
- **加密分享**：支持为 PRD 设置密码，通过 PBKDF2 + AES-GCM 加密后生成分享链接，接收方输入密码后方可解密查看，全程密码不经过服务器
- **平台同步**：支持将 PRD 一键同步到 Notion，飞书和语雀支持剪贴板复制
- **Issue 同步**：自动将 PRD 中的章节标题解析为 Issue，支持推送至 Linear 和 Jira

### 工程配套

- 单元测试覆盖核心 Store 和工具函数，使用 fake-indexeddb 模拟浏览器环境
